# Cross-Cutting Gotchas

- Contract counts drift across docs.
  Source truth today is:
  - 29 contract directories under `contracts/`
  - 28 genesis-deployed contracts in `genesis/src/lib.rs`
- The root `SKILL.md` is exhaustive and useful, but it is too large to be the default session bootstrap.
- Deployment docs and live handovers can disagree.
  Always compare exact dates before assuming a "current" network state.
- The worktree is often dirty with user edits or partially completed feature branches.
  Do not clean up unrelated files unless explicitly asked.
- Frontend shared helpers are synchronized copies.
  Treat `monitoring/shared/` as canonical and sync outward instead of patching many copies by hand.
- Normal Pages deploys should use `scripts/deploy-cloudflare-pages.sh`.
  Raw `wrangler pages deploy` is for CLI debugging only.
- Some commands intentionally skip local-private checks when the referenced path is missing.
  `scripts/run-local-private-check.sh` is the expected wrapper for that behavior.
- Local helper launchers are intentionally gated for dev use.
  Production or VPS paths should go through `deploy/setup.sh` and systemd-based flows.
- Fresh local full-stack validation must go through `scripts/start-local-stack.sh`, not just the 3-validator launcher.
  That path waits for `scripts/first-boot-deploy.sh`, which writes the signed metadata manifest and other post-genesis artifacts the DEX and portal control-plane reads depend on.
- Local frontends can show "missing contract addresses" even when the contracts exist on-chain if RPC is serving a live-synthesized signed metadata manifest with the wrong network name.
  On 2026-04-22 this surfaced as DEX failing to load `dex_core`, `dex_amm`, `dex_router`, `dex_margin`, `dex_rewards`, `dex_governance`, `dex_analytics`, and `prediction_market` until local RPC was forced back to the file-backed manifest generated by `first-boot-deploy.sh`.
- External/home validators can appear as ghost zero-stake peers if they are not publicly reachable on the expected P2P address.
  The observed live symptom was a valid announcement from `24.4.193.217:39129` for pubkey `8dkDiG...`, followed by repeated `Ignoring block range request from unknown peer` and reconnect timeouts; use the VPS/public-port runbook, not an ad-hoc NATed home setup, when validating external join flows.
- Live warp sync can fail even when checkpoint directories exist.
  On 2026-04-22 both US and EU testnet validators served checkpoint directories like `slot-677000`, but `latest_verified_checkpoint(...)` rejected them because `checkpoint_meta.json.state_root` did not match the committed block state root for the same slot. The likely source mismatch is that checkpoints are written after `apply_block_effects`, while block headers anchor the state root before those post-block effects.
- Warp checkpoint corroboration must be keyed by validator identity, not socket address.
  On 2026-04-23 the production pass found that counting `MIN_WARP_CHECKPOINT_ANCHOR_PEERS` by `SocketAddr` made checkpoint-root spoofing materially easier; the validator now only accepts checkpoint anchors from peers with a verified validator announcement and counts corroboration by validator `Pubkey`.
- VPS source trees can lag the local workspace even when they build and the validator starts.
  On 2026-04-22 US/EU `/home/ubuntu/lichen` lacked newer Solana index column-family code that existed locally and on SEA's `lichen-patched` tree; syncing only the touched validator files produced a binary drift that was still runnable on US/EU but could not open SEA's RocksDB.
- Do not stream validator binaries over raw `ssh ... cat ...` when SSH login banners are enabled.
  On 2026-04-22 the VPS legal banner contaminated a binary transfer and silently changed the file hash; use `scp -O` or another binary-safe transport instead.
- Local macOS release binaries are not deployable to the Linux VPS fleet.
  A direct install of the locally built `target/release/lichen-validator` on SEA failed with `Exec format error`; build the canonical deployment artifact on Linux.
- Historical audit and strategy docs can intentionally retain pre-publish package names.
  On 2026-04-23 the live README, developer portal, guides, and skill docs were updated to `@lobstercove/lichen-sdk`, `lobstercove-lichen-core`, `lichen-client-sdk = "0.1.0"`, and `cargo install --path cli/`, but dated audit/strategy snapshots were left unchanged unless they were still serving as active runbooks.
- RPC log-capture tests can race under parallel `cargo test --workspace` if they install independent tracing subscribers concurrently.
  On 2026-04-23 `rpc/src/lib.rs::capture_logs_async(...)` had to be serialized with a process-local mutex to keep privileged-audit-log assertions stable in the full workspace run.
- CI `Cargo Audit` checks every `Cargo.lock` in the repository, not just the workspace root.
  On 2026-04-23 the root lockfile audit was green while `compiler/Cargo.lock`, `sdk/Cargo.lock`, and many contract-local lockfiles still carried vulnerable `rand` versions; reproduce the workflow with `find . -path '*/target' -prune -o -name Cargo.lock -print | sort` and audit each lockfile explicitly.
- GitHub `Clippy` tracks the current stable toolchain, not whatever local stable happened to be last week.
  On 2026-04-23 local `rustc 1.94.0` was green while GitHub had already moved to `1.95.0` and surfaced new `unnecessary_sort_by`, `manual_checked_ops`, `collapsible_match`, and `unnecessary_cast` lints; if CI and local lint results disagree, run `rustup update stable` and validate with `cargo +stable clippy --workspace -- -D warnings`.
- Service-fleet testnet faucet probes currently depend on raw `9100/tcp` reachability for non-US VPSes.
  On 2026-04-23 the authoritative US `getServiceFleetStatus` probe was healthy locally on EU/SEA but timed out remotely because `deploy/setup.sh` removed `9100/tcp` while the generated service-fleet config still targeted `http://<public-ip>:9100/health`. The setup script now keeps `9100/tcp` open on testnet; if the probe model changes later, update the firewall policy and service-fleet config together.
- The US VPS build host can keep an old Rust toolchain active even after `rustup update stable`.
  On 2026-04-23 `rustup update stable` installed `1.95.0`, but the host still defaulted to `1.88.0` until `rustup default stable` was run and the build PATH was forced to `~/.cargo/bin`. If a remote build still claims an old `rustc` after an update, check `rustup show`, then explicitly switch the default toolchain before rebuilding.
- Monitoring frontend changes that add new RPC methods require a validator rollout, not just a Pages deploy.
  Mission Control runs against the validator-embedded RPC server from `lichen-validator`; on 2026-04-23 the frontend was ready locally, but `https://testnet-rpc.lichen.network` still returned `Method not found` for `getSporePumpStats` until the validator binary was rebuilt and rolled to US/EU/SEA.
- GitHub Release runs can fail on `actions/upload-artifact` even after the target binary and archive were built correctly.
  On 2026-04-23 the `v0.5.9` `x86_64-apple-darwin` release leg failed only on `CreateArtifact` timeouts inside `actions/upload-artifact@v4`; rerunning failed jobs on the same release run succeeded without any repo changes. If only artifact upload failed, prefer `gh run rerun <run-id> --failed` over cutting a new tag.
- Block cadence must not be derived from block-header Unix-second timestamps on a 400ms slot chain.
  On 2026-04-23 the old `average_block_time` path in `core/src/state/metrics_state.rs` and Mission Control produced fake `23%` production health because 400ms slots round into coarse second buckets. The durable fix is observer-side wall-clock telemetry (`observed_block_interval_ms`, `cadence_target_ms`, `head_staleness_ms`) plus cluster-level comparison from `getClusterInfo.cluster_nodes[].last_observed_block_slot`.
- Command-runner background reaping can make the local stack look flaky when it is not.
  On 2026-04-23 `scripts/start-local-stack.sh testnet` booted cleanly, but the validators disappeared once the parent shell exited because the command runner reaped background children. Keep the launcher shell alive (or use systemd/tmux) when validating the local 3-validator stack through this interface.
